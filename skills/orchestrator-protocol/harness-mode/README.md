# harness-mode — heavy-tier task pipeline

The optional "long-running project build" tier. Use when:

- The project has a BRD or multi-week roadmap.
- You want a live dashboard showing progress.
- Sessions span days/weeks and need continuity (the MANIFEST persists, individual orchestrator runs come and go).
- You want source-doc traceability (SD-### → task matrix).

Skip this tier for one-shot orchestrator runs — the light tier's `.orchestrator/tasks/{inbox,in-progress,done,escalated}/` is enough.

## Project shape under harness-mode

```
<project-root>/
├── docs/
│   ├── BRD.docx + BRD.md + BRD-INDEX.md
│   ├── existing-app-profile.md
│   ├── defaults.md                       # DEF-NNN registry
│   ├── source-doc-coverage.md            # SD-### → task matrix
│   ├── open-questions.md
│   └── adr/
├── modules/
│   └── <feature-name>/README.md          # per-feature decomposition
├── harness/
│   ├── README.md
│   ├── protocols/
│   │   ├── lifecycle.md                  # state machine + DoD + blocked recovery
│   │   ├── quality.md                    # review + polish + qa gates
│   │   └── git.md                        # checkpoint + rollback contract
│   ├── tasks/
│   │   ├── MANIFEST.json                 # canonical task list
│   │   ├── MANIFEST.schema.json
│   │   ├── TASK_TRACKER.md               # human-readable mirror
│   │   ├── backlog/TASK-NNNN.md
│   │   ├── in-progress/TASK-NNNN.md      # multiple files here = parallel work
│   │   ├── review/TASK-NNNN.md           # post code-review gate
│   │   ├── polish/TASK-NNNN.md           # post UI polish gate (skipped if ui_weight=none)
│   │   ├── qa/TASK-NNNN.md               # post QA gate
│   │   ├── done/TASK-NNNN.md
│   │   └── blocked/TASK-NNNN.md
│   ├── state/
│   │   ├── DASHBOARD.md                  # regenerated by scripts/status.sh
│   │   ├── SESSION_LOG.md                # append-only
│   │   ├── CHECKPOINTS.md                # every checkpoint/* tag
│   │   ├── OPEN_QUESTIONS.md
│   │   ├── DECISIONS_LOG.md
│   │   └── .orchestrator.lock            # gitignored; same lock as .orchestrator/session.lock
│   ├── templates/                        # task-template.md, ADR template, report templates
│   ├── qa/
│   │   ├── playwright.config.ts
│   │   └── specs/
│   └── scripts/                          # status.sh / pick.sh / move.sh / checkpoint.sh / rollback.sh
└── .orchestrator/                        # still exists; light-tier queue for ad-hoc work
```

The `.orchestrator/` queue and `harness/tasks/` queue coexist:

- `.orchestrator/tasks/` is for ad-hoc, non-MANIFEST work (a one-off ticket from log-reader-triage, a finding the PR reviewer files).
- `harness/tasks/` is the project's persistent backlog generated by `task-architect`.

The orchestrator drains BOTH in each wave.

## Promoting light → harness

```
/orchestrate promote this project to harness-tier
```

Spawns `task-architect`. The agent reads the BRD/modules/existing-app-profile and produces the harness layout. Existing `.orchestrator/` tickets stay where they are.

## Dashboard

Harness Snitch is an **external, not-shipped** reference dashboard (any filesystem-polling status viewer works). Point your own at any framework-compliant project:

```
HARNESS_ROOT=/path/to/<project>/harness npm --prefix /path/to/harness-snitch run dev
```

The Snitch reads `harness/tasks/MANIFEST.json`, the seven stage folders, and `harness/state/CHECKPOINTS.md`. It supports parallel in-flight tasks natively because it just iterates `in-progress/`.

## Files in this directory

- `protocols/lifecycle.md` — the task state machine (backlog → in-progress → review → polish → qa → done | blocked).
- `protocols/quality.md` — the three review gates (code-review, polish, QA).
- `protocols/git.md` — checkpoint + rollback contract.
- `templates/MANIFEST.schema.json` — JSON schema for the manifest.
- `templates/MANIFEST.example.json` — small example manifest.
- `templates/task.md.tmpl` — per-task file template.
- `templates/release-notes.md.tmpl` — output of `release-manager.yaml`.
