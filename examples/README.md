# examples/ — one tiny slice flowing through every stage

The kit narrates the **BRD → ingest → contract → build → integrate → QA** loop thoroughly, but a
reader needs to *see* the artifacts each stage produces. This is one small-but-complete vertical
slice — a **Room Booking** feature (request a room, an approver confirms/rejects) — carried through
the whole loop. It is deliberately tiny and synthetic; the point is to show the *shape* of "done" at
each stage, not an exhaustive app.

Walk it in order:

| Stage | Folder | What it shows |
|---|---|---|
| 0. **Corpus** | [`walkthrough/00-corpus/`](walkthrough/00-corpus/) | The synthetic source docs (stand-ins for `.docx`/`.pdf`), with native requirement IDs `BKG-BR-*` / `BKG-AUTH-*`. |
| 1. **Knowledge base** | [`walkthrough/01-kb/`](walkthrough/01-kb/) | What `ingest.py` + `gen_index.py` generate: `INDEX.md`, a converted `.md` carrying its `> Source:` line, a curated `_manifest.md`, and `INGEST-MANIFEST.tsv`. **Really generated by the tooling** — re-run the command below to reproduce it byte-for-byte. |
| 2. **Development contract** | [`walkthrough/02-contract/`](walkthrough/02-contract/) | A filled `acceptance-criteria.md` ledger (note the **Verification ref** column — tests for protected-core ACs, `live-app:` for the rest), a filled `design-spec.md`, and a filled `build-plan.md`. |
| 3. **Tickets** | [`walkthrough/03-tickets/`](walkthrough/03-tickets/) | Two real tickets: one in [`done/`](walkthrough/03-tickets/done/) with a `## Resolution`, one in [`escalated/`](walkthrough/03-tickets/escalated/) with a `## Escalation reason`. The lifecycle is the **directory**, not a status field. |
| 4. **Reports** | [`walkthrough/04-reports/`](walkthrough/04-reports/) | A sample `INTEGRATION-REPORT.md` (the separate `/kit-integrate` run) and a `QA-FINDINGS-approver.md` showing the Sonnet tester output **and** the mandatory Phase-3 adversarial re-verification. |

## Reproduce stage 1 yourself

The KB under `01-kb/` was produced by the real ingestion tooling on the corpus under `00-corpus/`
(markdown originals pass through, so no markitdown/extractor venv is needed for this example):

```bash
python3 toolkit/ingestion/ingest.py    examples/walkthrough/00-corpus examples/walkthrough/01-kb
python3 toolkit/ingestion/gen_index.py examples/walkthrough/01-kb --write-manifests
python3 toolkit/ingestion/gen_index.py examples/walkthrough/01-kb --check   # the stage-1 → stage-2 gate
```

## How the protected core shows up here

This tiny slice is mostly **protected core**, which is exactly why it's a good teaching example for the
[testing policy](../PLAYBOOK.md) (§7): the guarded **state machine** (AC-BKG-01), the **temporal**
no-overlap rule (AC-BKG-02), the **RBAC** confirm/reject gate + read-scope (AC-BKG-03/04), and the
append-only **audit** (AC-BKG-05) each carry a real test against a real DB. The two non-core ACs —
the notification UX (AC-BKG-06) and list rendering (AC-BKG-07) — are verified on the live app, no test.

Read [`../PLAYBOOK.md`](../PLAYBOOK.md) for the narrated method this slice instantiates.
